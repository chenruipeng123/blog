## 问题背景

某天下午，生产环境的几个微服务容器开始随机重启。`docker ps -a` 显示退出状态为 `OOMKilled`（Exit Code 137）。

## 排查步骤

### 1. 查看容器资源使用

```bash
docker stats --no-stream
```

发现某个 Java 服务内存使用持续上涨，远超 `-Xmx` 限制。

### 2. 检查 JVM 参数

```dockerfile
# 原来的配置
ENV JAVA_OPTS="-Xmx512m -Xms256m"
```

问题在于：JVM 的 `-Xmx` 只限制堆内存，但 JVM 还使用堆外内存（Metaspace、线程栈、Direct Buffer 等）。

### 3. 确认根因

容器设置了 `--memory=768m`，但 JVM 实际使用：
- 堆内存：最大 512MB
- Metaspace：~100MB
- 线程栈：40 个线程 × 1MB = 40MB
- Direct Buffer：~50MB
- JVM 自身开销：~50MB
- **合计：~752MB，接近 768MB 上限**

在 GC 来不及回收时，容器就会被 OOM Killer 杀掉。

## 解决方案

```dockerfile
# 修改后
ENV JAVA_OPTS="-Xmx384m -Xms256m \
  -XX:MaxMetaspaceSize=128m \
  -XX:MaxDirectMemorySize=64m \
  -XX:ReservedCodeCacheSize=64m"
```

给 JVM 留出足够的内存余量，堆内存从 512MB 降到 384MB。

## 总结

> 容器内存限制 ≠ JVM 堆内存。JVM 进程使用的总内存大约是：堆 + Metaspace + 线程栈 × 线程数 + Direct Buffer + CodeCache + 其他。建议给容器预留 25%-30% 的非堆内存空间。
